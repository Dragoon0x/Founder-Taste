# Empty States

An empty screen is not a bug. It's a design opportunity. Treat it like one.

## How to use

- `/empty-states` Design empty states for any product in this conversation.

## Constraints

### What Every Empty State Needs
1. **Explanation:** What will appear here once the user takes action. "This is where your projects will live."
2. **Action:** A clear CTA to create the first item. "Create Your First Project" button, not just a + icon.
3. **Motivation:** Why this action is worth doing. "Projects help you organize your work and track progress."
- Optional: an illustration or icon that makes the state feel intentional, not broken.
- MUST make the first-action CTA the most prominent element on the empty screen.

### Where Empty States Live
- Dashboard on first login (no data yet)
- Any list/table with zero items
- Search results with no matches
- Filtered views with no results
- Inbox/notification center with nothing new
- MUST design every one of these. Default blank screens feel like bugs.

### The First-Use vs. Cleared Distinction
- First-use empty (never had data): encouraging, educational, action-oriented. "Create your first project to get started."
- Cleared empty (had data, now empty): informational, calm. "No tasks due today. Nice work." or "No results match your filters. Try adjusting."
- MUST use different copy for these two states. A returning user doesn't need to be told what projects are.

### Anti-Patterns
- Blank white screens with no guidance
- "No data" as the only message
- Empty states that look like error states
- Overly cute illustrations without a clear action
- Empty states that make users feel like they did something wrong
